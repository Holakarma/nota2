# Документация проекта Nota

## Назначение проекта

Nota - фронтенд-приложение для работы с заметками, потоками и чатовым вводом заметок. Пользователь может регистрироваться, входить в аккаунт, создавать потоки, отправлять сообщения в чат, получать созданные системой заметки и потоки, просматривать похожие заметки и редактировать содержимое заметок в markdown-редакторе.

Проект находится в директории `frontend` и реализует клиентскую часть приложения. Серверная часть подключается через REST API, типы и API-клиент генерируются из OpenAPI-спецификации.

## Основные технологии

### Базовый стек

- **React 19** - основная библиотека для построения интерфейса.
- **TypeScript 6** - статическая типизация, строгая проверка проекта через `tsc -b`.
- **Vite 8** - dev-сервер, сборка и инфраструктура фронтенда.
- **React DOM** - рендеринг приложения в браузере.
- **ES2023** - целевая версия JavaScript в `tsconfig.app.json`.

### UI и стилизация

- **Material UI (`@mui/material`)** - компоненты интерфейса: `Button`, `TextField`, `Card`, `Stack`, `Grid`, `Snackbar`, `Alert`, `Chip`, `Tooltip`, `CircularProgress` и другие.
- **Emotion (`@emotion/react`, `@emotion/styled`)** - CSS-in-JS слой, используемый Material UI.
- **Кастомная тема MUI** - создается в `src/app/providers/theme/theme.ts`.
- **Светлая и темная тема** - режим хранится в `localStorage` через Zustand.
- **Шрифты Rubik и Molle** - подключаются из `public/fonts` через `MuiCssBaseline`.
- **Кастомные варианты типографики** - `M40`, `R20`, `L16`, `Logo`, `NoteMarkdownH1` и другие определены в `src/app/providers/theme/config/typography.ts`.

### Роутинг

- **TanStack Router** - типобезопасная маршрутизация.
- Используются публичные маршруты только для неавторизованных пользователей и защищенные маршруты для основной части приложения.
- Защита маршрутов выполняется через проверку access token и попытку обновления токена refresh-запросом.
- Компоненты страниц загружаются лениво через `lazyRouteComponent`.

### Работа с серверным состоянием

- **TanStack Query** - загрузка данных, кеширование, мутации, инвалидация списков и деталей.
- Для списков предусмотрена cursor-пагинация.
- Для некоторых списков используется `keepPreviousData`, чтобы интерфейс не мигал при смене параметров.
- Для сообщений используется infinite query и виртуализированный список.

### Формы и валидация

- **React Hook Form** - управление формами входа и регистрации.
- **Zod** - валидация пользовательского ввода и API-параметров.
- **@hookform/resolvers/zod** - интеграция Zod-схем с React Hook Form.

### HTTP и API

- **Axios** - HTTP-клиент.
- **swagger-typescript-api** - генерация TypeScript API-клиента из OpenAPI-файла `swagger/api.swagger.json`.
- Сгенерированный клиент расположен в `src/shared/api/generated`.
- Модуль `src/shared/api/client.ts` собирает API-клиенты `auth`, `chat`, `note`, `stream`.
- Access token хранится в памяти приложения, refresh token ожидается в cookie, так как HTTP-клиенты настроены с `withCredentials: true`.
- При ответе `401` приложение автоматически пытается обновить access token через `/api/auth/refresh` и повторить исходный запрос.

### Локальное состояние

- **Zustand** - локальные хранилища:
  - `theme-mode` - выбранная тема интерфейса;
  - `stream-message-draft` - черновик сообщения в поле ввода.
- Оба хранилища используют persist middleware и `localStorage`.

### Производительность интерфейса

- **@tanstack/react-virtual** - виртуализация списка сообщений.
- Кастомные хуки resize/intersection observer помогают подгружать старые сообщения и удерживать корректный scroll-позиционинг.
- Поисковые запросы throttling-ятся перед отправкой на сервер.

### Инструменты разработки

- **ESLint 10** - проверка JavaScript/React-кода.
- **Prettier plugin для ESLint** - контроль форматирования.
- **npm scripts**:
  - `npm run start` - запуск Vite dev-сервера;
  - `npm run build` - TypeScript-проверка и production-сборка;
  - `npm run lint` - запуск ESLint;
  - `npm run preview` - предпросмотр production-сборки;
  - `npm run codegen` - генерация API-клиента из OpenAPI.

## Конфигурация окружения

Приложение использует переменные окружения Vite:

- `VITE_API_BASE_URL` - базовый URL API. Значение по умолчанию: `http://localhost:3000`.
- `VITE_API_REFRESH_PATH` - путь refresh-запроса. Значение по умолчанию: `/api/auth/refresh`.

Файл `.env` находится в корне frontend-проекта.

## Архитектура проекта

Структура проекта близка к Feature-Sliced Design:

- `src/app` - инициализация приложения, провайдеры, тема, роутер, общий layout.
- `src/pages` - страницы приложения и их route-фабрики.
- `src/widgets` - крупные самостоятельные блоки интерфейса: header, sidebar потоков, заголовок потока.
- `src/features` - пользовательские действия и сценарии: авторизация, отправка сообщений, поиск.
- `src/entities` - бизнес-сущности и их API-слой: user, note, stream, chat.
- `src/shared` - общие API-клиенты, UI-компоненты, иконки, утилиты, модели и сгенерированный код.

В проекте настроены алиасы:

- `@app`
- `@pages`
- `@widgets`
- `@features`
- `@entities`
- `@shared`

## Возможности интерфейса

### Авторизация

На странице входа пользователь может:

- ввести логин и пароль;
- пройти клиентскую валидацию полей;
- отправить форму входа;
- перейти к регистрации;
- увидеть серверные ошибки через snackbar;
- увидеть кнопку входа через Telegram.

Кнопка Telegram присутствует в интерфейсе, но в текущей реализации у нее не подключен обработчик авторизации.

На странице регистрации пользователь может:

- ввести логин;
- ввести пароль;
- повторить пароль;
- получить ошибку, если пароли не совпадают;
- отправить форму регистрации;
- перейти на страницу входа;
- увидеть ошибки сервера через snackbar.

Правила валидации:

- логин: от 8 до 255 символов;
- пароль: от 8 до 128 символов;
- повтор пароля должен совпадать с паролем.

После успешного входа или регистрации пользователь перенаправляется к списку заметок.

### Защищенная оболочка приложения

После авторизации интерфейс отображается в общем layout:

- верхняя панель высотой 60px;
- логотип `Nota`;
- переключатель светлой/темной темы;
- логин текущего пользователя;
- основная область с максимальной шириной 1440px;
- адаптивная сетка для sidebar и основного содержимого.

Если access token отсутствует, приложение пытается обновить его через refresh-запрос. Если refresh не удается, пользователь перенаправляется на `/login`.

### Потоки

Потоки используются как тематические группы заметок и отдельные контексты чата.

В боковой панели потоков пользователь может:

- открыть общий список через пункт `Все`;
- создать новый поток через `+ Новый поток`;
- ввести название потока в браузерном prompt;
- перейти к конкретному потоку;
- видеть выбранный поток по активному состоянию элемента списка.

Список потоков загружается с лимитом 20 элементов.

### Список заметок

Страница `/stream` и `/stream/:streamId` показывает заметки карточками.

Пользователь может:

- просматривать все заметки;
- просматривать заметки конкретного потока;
- видеть название выбранного потока в верхней части экрана;
- перейти из списка заметок в чат текущего контекста;
- открыть заметку по карточке;
- видеть состояние загрузки;
- видеть ошибку, если заметки не удалось загрузить или идентификатор потока некорректный.

Карточка заметки показывает:

- короткий preview-текст;
- названия связанных потоков;
- обрезку длинного текста до нескольких строк.

### Поиск похожих заметок в списке

Поле сообщения внизу страницы потока одновременно работает как поисковый ввод.

Когда пользователь вводит текст:

- текст сохраняется как черновик;
- запрос нормализуется и ограничивается 500 символами;
- поиск throttling-ится на 700 мс;
- список заметок переключается на режим похожих заметок;
- сервер возвращает до 50 похожих заметок.

Если строка поиска пустая, отображается обычный список заметок.

### Чат

Страница `/chat` и `/chat/:streamId` реализует чатовый сценарий создания заметок.

Пользователь может:

- открыть общий чат;
- открыть чат конкретного потока;
- видеть историю сообщений;
- отправлять сообщение кнопкой отправки;
- отправлять сообщение клавишей `Enter`;
- переносить строку через `Shift + Enter`;
- видеть состояние пустого чата;
- видеть состояние загрузки сообщений;
- видеть ошибку, если чат или поток недоступны.

Перед отправкой приложение получает существующий чат или создает его через ensure-запрос. После отправки сообщение добавляется в кеш списка сообщений.

### Результаты обработки сообщения

Каждое сообщение отображается в двух частях:

- пользовательское сообщение справа;
- системный результат под ним, если сервер вернул результат обработки.

Системный результат может включать:

- карточку созданной заметки;
- список созданных потоков;
- ссылки на заметку или поток.

Если заметка открыта из чата потока, маршрут сохраняет контекст возврата обратно в чат.

### Поиск похожих заметок и потоков в чате

Под историей сообщений отображается блок похожих результатов для текущего черновика сообщения.

Поведение поиска:

- обычный текст ищет похожие заметки;
- текст с префиксом `:` ищет похожие потоки;
- запрос ограничивается 500 символами;
- запрос throttling-ится на 500 мс;
- результаты отображаются горизонтальным списком карточек;
- ошибки поиска показываются через snackbar.

Пример:

- `идея диплома` - поиск похожих заметок;
- `:работа` - поиск похожих потоков.

### Поле ввода сообщения

Общий компонент `MessageInput` используется в списке заметок и в чате.

Возможности:

- многострочный ввод;
- ограничение до 32768 символов;
- отправка по `Enter`;
- перенос строки по `Shift + Enter`;
- кнопка отправки с иконкой;
- блокировка при пустом вводе;
- блокировка во время отправки;
- сохранение черновика в `localStorage`;
- очистка черновика после успешной отправки.

### Редактор заметки

Страница заметки позволяет просматривать, редактировать и удалять заметку.

Пользователь может:

- открыть заметку из общего списка, списка потока или чата;
- вернуться назад в соответствующий контекст;
- удалить заметку;
- редактировать markdown-содержимое;
- видеть статус синхронизации;
- управлять привязанными потоками заметки.

Если заметка не найдена или контекст маршрута некорректный, отображается экран `Заметка не найдена` с кнопкой возврата к заметкам.

### Markdown-редактор

Редактор заметки реализован построчно:

- неактивные строки показываются как preview;
- при клике или фокусе строка превращается в поле ввода;
- `Enter` разделяет строку;
- `Backspace` в начале строки объединяет ее с предыдущей;
- `Delete` в конце строки объединяет ее со следующей;
- стрелки вверх/вниз перемещают фокус между строками;
- вставка многострочного текста корректно разбивается на строки;
- длина заметки ограничена 32768 символами.

Поддерживаемые элементы markdown-превью:

- заголовки `#` - `######`;
- обычные абзацы;
- пустые строки;
- маркированные списки через `-`;
- нумерованные списки через `1.`;
- fenced code blocks через тройные backtick;
- inline code через backtick;
- выделение `*bold*`;
- курсив `_italic_`;
- экранирование markdown-символов через `\`.

### Автосинхронизация заметки

Содержимое заметки синхронизируется с сервером автоматически.

Поведение синхронизации:

- изменения throttling-ятся на 1000 мс;
- при уходе фокуса из редактора выполняется принудительная попытка сохранения;
- пустое содержимое не синхронизируется;
- повторная отправка одинакового неуспешного содержимого не выполняется до нового изменения;
- при успешном сохранении инвалидируются списки и похожие заметки.

Индикатор статуса показывает:

- `Синхронизация недоступна`;
- `Синхронизация...`;
- `Заметка синхронизирована`;
- `Не удалось синхронизировать заметку`.

### Управление потоками заметки

В верхней части редактора заметки отображаются привязанные потоки.

Пользователь может:

- видеть потоки как chips;
- отвязать поток от заметки через кнопку удаления chip;
- ввести название потока;
- добавить заметку в существующий поток;
- создать новый поток и сразу привязать его к заметке;
- получить ошибку, если поток уже привязан;
- получить ошибку, если название слишком длинное;
- получить snackbar при неудачной привязке или отвязке.

Название потока ограничено 128 символами. Перед сравнением названия нормализуются: обрезаются пробелы, повторные пробелы схлопываются, строка приводится к нижнему регистру.

### Страница профиля

Маршрут `/profile` существует, но интерфейс страницы пока является заглушкой с текстом `Profile page`.

### Ошибки и уведомления

В приложении используется общий `SnackbarProvider`.

Сейчас через snackbar показываются:

- ошибки входа;
- ошибки регистрации;
- ошибки поиска похожих заметок;
- ошибки поиска похожих потоков;
- ошибки добавления или удаления связи заметки с потоком;
- ошибки валидации некоторых пользовательских действий.

Snackbar отображается в правом нижнем углу и автоматически скрывается через 5 секунд.

## API-возможности, используемые интерфейсом

### Auth

- `POST /api/auth/sign-up` - регистрация.
- `POST /api/auth/sign-in` - вход.
- `POST /api/auth/refresh` - обновление токенов.
- `POST /api/auth/sign-out` - выход, клиентский UI сейчас явно не использует эту возможность.
- `GET /api/auth/me` - получение текущего пользователя.

### Note

- `POST /api/note` - создание заметки.
- `GET /api/note` - список заметок.
- `GET /api/note/similar` - поиск похожих заметок.
- `GET /api/note/:id` - получение заметки.
- `PATCH /api/note/:id` - обновление заметки.
- `DELETE /api/note/:id` - удаление заметки.

### Stream

- `POST /api/stream` - создание потока.
- `GET /api/stream` - список потоков.
- `GET /api/stream/similar` - поиск похожих потоков.
- `GET /api/stream/:id` - получение потока.
- `PATCH /api/stream/:id` - обновление потока, API-слой поддерживает, отдельный UI редактирования названия сейчас не реализован.
- `DELETE /api/stream/:id` - удаление потока, API-слой поддерживает, отдельный UI удаления сейчас не реализован.
- `GET /api/stream/:streamId/notes` - заметки потока.
- `POST /api/stream/:streamId/notes/:noteId` - привязка заметки к потоку.
- `DELETE /api/stream/:streamId/notes/:noteId` - отвязка заметки от потока.

### Chat

- `POST /api/chat` - создание чата, API-слой поддерживает.
- `PUT /api/chat` - получить существующий или создать чат.
- `GET /api/chat` - список чатов, API-слой поддерживает.
- `GET /api/chat/:chatId/messages` - сообщения чата.
- `POST /api/chat/:chatId/messages` - отправка сообщения.

## Работа с данными и кешем

Для каждой сущности описаны query keys и mutation hooks:

- `noteQueries`, `useCreateNoteMutation`, `useUpdateNoteMutation`, `useRemoveNoteMutation`;
- `streamQueries`, `useCreateStreamMutation`, `useUpdateStreamMutation`, `useRemoveStreamMutation`, `useAttachNoteToStreamMutation`, `useDetachNoteFromStreamMutation`;
- `chatQueries`, `useCreateChatMutation`, `useEnsureChatMutation`, `useCreateChatMessageMutation`;
- `currentUserQueries`.

После мутаций приложение инвалидирует связанные кеши:

- после изменения заметки обновляются детали, списки и похожие заметки;
- после создания сообщения обновляются списки чатов и потоков;
- после изменения потоков обновляются списки, детали и связанные заметки;
- после привязки или отвязки потока от заметки обновляется заметка и списки заметок.

## Сгенерированный API-код

API-клиент генерируется командой:

```bash
npm run codegen
```

Генератор берет OpenAPI-спецификацию из:

```text
swagger/api.swagger.json
```

Результат сохраняется в:

```text
src/shared/api/generated
```

Генерация настроена в `codegen/codegen.js`:

- HTTP-клиент: axios;
- режим: modular;
- request body, query/path params и response body выносятся в отдельные типы;
- клиент разбивается на модули `Auth`, `Chat`, `Note`, `Stream`.

## Текущие ограничения интерфейса

- Страница профиля пока является заглушкой.
- Кнопка Telegram на странице входа отображается, но не запускает авторизацию.
- UI выхода из аккаунта не реализован, хотя API-клиент содержит метод `sign-out`.
- API-слой поддерживает обновление и удаление потоков, но в интерфейсе доступны только создание потока и привязка/отвязка потоков к заметке.
- В зависимостях установлены `stream-chat` и `stream-chat-react`, но текущий интерфейс чата использует собственный REST API-клиент и компоненты проекта.
